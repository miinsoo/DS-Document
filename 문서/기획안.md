## MQTT를 이용한 M:N 통신
### 문제 인식
- Tight Coupling: 1대 1 직접 통신 환경에서는 장비 한 대가 늘어날 때마다 서버의 소켓 관리 부하가 선형적으로 증가하며, 장비와 서버 간의 높은 의존성으로 인해 확장성이 제한됩니다.
- 비용 및 진입 장벽: Interface A, SECS/GEM, HSMS 등 산업 표준은 높은 라이선스 비용과 폐쇄적인 SDK 구조로 인해 개발 접근성에 제약이 큽니다.
- 성능적 한계: 텍스트 기반의 HTTP/SOAP(XML) 통신은 Payload가 무거워 대용량 데이터 전송 시 높은 네트워크 오버헤드와 처리 지연을 초래합니다.
- 파편적인 데이터 형식: XML, SECS Message으로 파편화된 데이터 형식으로 현대의 웹과의 연동이 힘들어집니다.

### 해결방안
- MQTT 기반 Pub/Sub 오케스트레이션: 브로커를 중재자로 활용하여 공장 설비(EAP)와 다수의 클라이언트(MES, Historian, 모바일) 간의 Loose Coupling을 실현하고 M:N 통신을 구현합니다.
- 전략적 프로토콜 현대화: 산업 표준의 핵심 기능(이벤트 보고, 알람 관리)은 구현하되, 구현체는 MQTT + JSON 조합으로 경량화하여 웹/모바일 생태계와의 결합력을 높입니다.
### 필요한 서버 종류
- Broker 서버
    - 설명: 비전 기기 데이터를 모바일 기기와 수집 서버에게 전달
    - 데이터 흐름: 가상 EAP 서버(비전 머신) ==> Broker 서버 ==> 모바일 기기, 수집 서버
    - 오픈소스: Eclipse Mosquitto
- Historian 서버
    - 설명: 비전 기기 데이터를 TSDB(시계열 데이터베이스)에 적제 
    - 데이터 흐름: 가상 EAP 서버(비전 머신) ==> Broker 서버 ==> Historian 서버   
    - 플렛폼: Node.js(TypeScript)
    - DB: Postgres TimescaleDB(TBMS), InfluxDB
- 가상 EAP 서버
    - 설명: 비전 기기 데이터를 Json 포멧으로 데이터 발행
    - 언어: C#
- MES 서버
    - 설명: 비전 기기를 MQTT로 제어(기기 상태 변환, 레시피 변환) 밎 lot end 정보를 저장
    - 데이터 흐름: MES <==> Broker(MES 내부) <==> 가상 EAP 서버
    - 언어: C#
- Dispatcher 서버
    - 설명: TSDB(시계열 데이터베이스)에서 read-only로 데이터를 조회 후, AI 서버로 데이터를 전송(push)
    - 데이터 흐름: TSMS ⇒ Dispatcher 서버 ⇒ AI 서버
    - 플렛폼: Node.js(TypeScript)

## LLM을 이용한 자동 리서치 보고서 작성
### 문제 인식
- 통합 관제 가시성 결여: 특정 구역 내 다수 설비에서 발생하는 상호 연관된 퍼포먼스 지표와 이벤트를 단일 관점에서 통합 모니터링하고 분석할 수 있는 고도화된 솔루션이 부재합니다.
- 분석 생산성의 한계: EDA 등을 통해 수집되는 방대한 시계열 및 로그 데이터를 소수의 분석가가 수동으로 분석하는 방식은 막대한 시간과 리소스가 소요되며, 실시간 이상 징후 포착 및 예지 보전에 한계가 있습니다.
- 데이터 분석의 병목 현상: 수만 건의 비정형 로그 속에서 유의미한 상관관계를 찾아내는 과정에서 분석가의 숙련도에 따라 통찰의 질이 결정되는 인적 의존성 문제가 발생합니다.

### 해결 방안
- RAG(검색 증강 생성) 기반 지능형 분석: Historian 서버(TSDB)에 적재된 시계열 로그와 설비 매뉴얼을 벡터 데이터베이스에 임베딩하여, 현재 구역의 상태를 다각도로 분석하는 RAG 파이프라인을 구축합니다.
- LLM 기반 자동 리서치 리포팅: 복잡한 수치와 텍스트 로그를 LLM이 실시간으로 요약 및 분석하여, 구역의 효율성 리포트와 권고안을 인간이 이해하기 쉬운 자연어 형태로 자동 생성합니다.
- 의사결정 최적화: 분석가는 원시 데이터를 뒤지는 대신 AI가 생성한 인사이트를 검토하고 최종 의사결정만 내림으로써, 공정 관리의 효율성을 극대화합니다.
### 필요한 서버 종류
- AI 서버
    - 설명: Historian 서버에서 데이터를 받아(push), 전처리 -> 임베딩 -> 적제. 웹 백엔드 서버에서 정보 요청 시, 데이터를 전달
    - 데이터 흐름: Historian 서버 ==> AI 서버 ==> 웹 백엔드 서버
    - 플렛폼: Python
    - DB: Postgres pgvector(Vector DB), Chroma DB
- 웹 백엔드 서버
    - 설명: 프론트 서버와 REST API로 통신, 자동 리서치 보고서에 필요한 정보를 제공
    - 데이터 흐름: AI 서버 ==> 웹 벡엔드 서버 ==> 프론트 서버
    - 플렛폼: Spring Boot(Java)
    - DB: Postgres Core(RDBMS)
