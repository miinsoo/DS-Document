## MQTT를 이용한 M:N 통신
### 문제 인식
- Tight Coupling: 1대 1 직접 통신 환경에서는 장비 한 대가 늘어날 때마다 서버의 소켓 관리 부하가 선형적으로 증가하며, 장비와 서버 간의 높은 의존성으로 인해 확장성이 제한됩니다.
- 비용 및 진입 장벽: Interface A, SECS/GEM, HSMS 등 산업 표준은 높은 라이선스 비용과 폐쇄적인 SDK 구조로 인해 개발 접근성에 제약이 큽니다.
- 성능적 한계: 텍스트 기반의 HTTP/SOAP(XML) 통신은 Payload가 무거워 대용량 데이터 전송 시 높은 네트워크 오버헤드와 처리 지연을 초래합니다.
- 파편적인 데이터 형식: XML, SECS Message으로 파편화된 데이터 형식으로 현대의 웹과의 연동이 힘들어집니다.
- Lot 단위 사후 점검의 인적 병목 현상: 개별 Lot 작업이 완료된 직후, 결과물의 결함 여부와 설비 상태를 확인하는 과정이 작업자의 수동 검토에 의존하고 있습니다. 이는 데이터가 방대해질수록 분석 속도가 저하되는 병목 현상을 야기하며, 특정 Lot에서 발생한 미세한 이상 징후가 즉각적으로 피드백되지 않아 설비 점검의 골든타임을 놓칠 위험이 존재합니다.
- 경직성 및 장애 전이: 모든 기능을 단일 서버에 집중할 경우, 특정 로직(예: 분석 엔진)의 오류가 전체 공정 제어 시스템의 중단으로 이어지는 리스크가 큽니다. 또한, 기능 추가 시 기존 코드와의 간섭으로 인해 유지보수 효율이 저하됩니다.
- 구조적 유연성 부재: 장비와 서버 간의 강한 의존성은 새로운 데이터 소비자(예: 신규 모바일 앱, 외부 AI 서버)가 추가될 때마다 전체 네트워크 구조를 재설계해야 하는 비효율을 초래합니다.

### 해결방안
- MQTT 기반 Pub/Sub 오케스트레이션: 브로커를 중재자로 활용하여 공장 설비(EAP)와 다수의 클라이언트(MES, Historian, 모바일) 간의 Loose Coupling을 실현하고 M:N 통신을 구현합니다.
- 전략적 프로토콜 현대화: 산업 표준의 핵심 기능(이벤트 보고, 알람 관리)은 구현하되, 구현체는 MQTT + JSON 조합으로 경량화하여 웹/모바일 생태계와의 결합력을 높입니다.
- 하이브리드 분석 로직
    - 정량적 1차 판정 (Rule-based): BGA 볼 개수, 치핑 크기, 패키지 치수 등 정밀 검사 파라미터를 사전에 정의된 임계치와 대조합니다. 이를 통해 규격을 벗어난 제품을 실시간으로 분류하고 양/불 판정을 확정합니다.
    - 지능형 이상 징후 포착 (Local AI): 수치상으로는 임계치 이내에 있더라도, 데이터 간의 비정형적 패턴을 로컬 AI가 실시간으로 분석합니다. 단순 룰로 잡아내기 어려운 잠재적 결함이나 추가 점검이 필요한 결과물을 선별하여 즉각적인 알림을 생성하여 예지 보존을 도모합니다.
- 단일 책임 원칙 기반의 서버 오케스트레이션: 각 서버가 하나의 명확한 역할(Historian-저장, Oracle-판정, Dispatcher-보안 전송, AI-분석)만 수행하도록 분리하여 시스템의 응집도를 극대화합니다. 이를 통해 분리된 서버들이 메시지 스트림을 통해 유기적으로 연결되어, 특정 분석 서버에 장애가 발생하더라도 공정 제어와 데이터 적재는 독립적으로 유지되는 결함 격리(Fault Isolation) 구조를 구축합니다.
- 무중단 확장: 메시지 브로커를 중재자로 활용하여, 새로운 분석 엔진이나 모니터링 클라이언트는 기존 시스템의 중단 없이 브로커의 토픽을 구독하는 것만으로 즉시 연동되는 플러그 앤 플레이 방식의 확장을 실현합니다.

### 필요한 서버 종류
- Broker 서버
    - 설명: MQTT Pub/Sub 모델의 중재자 역할을 하는 메시지 브로커 서버로, EAP(Publisher)와 다수의 Subscriber(모바일, Oracle, Historian) 사이의 메시지 전달을 담당합니다.
    - 목적: 공장 설비(EAP)와 다수의 클라이언트 간의 Loose Coupling M:N 통신을 실현합니다. 장비와 서버가 늘어나도 Subscriber를 추가하기만 하면 되므로 높은 확장성을 보임
    - 필요성
        - 1:1 직접 소켓 통신은 장비 수 증가 시 서버의 소켓 관리 부하가 선형적으로 증가하는 Tight Coupling 문제가 있어 MQTT 기반 구조로 해소
        - 망 분리 환경에서 외부 인터넷 없이 현장 내부 Wi-Fi만으로 M:N 통신을 구현하기 위해 필요
        - 모바일 앱 Wi-Fi 단절 시 QoS 설정에 따라 재연결 후 메시지를 재전달하여 고가용성 확보
    - 데이터 흐름: 가상 EAP 서버(비전 머신) ==> Broker 서버 ==> 모바일 기기, 
    - 방식: Eclipse Mosquitto (오픈소스)
- Historian 서버
    - 설명: Broker에서 수신한 비전 검사 결과 데이터를 시계열 데이터베이스(TSDB)에 적재하고 보존하는 데이터 수집 서버
    - 목적: EAP에서 발행되는 모든 검사 결과를 시계열 형태로 장기 보존하여 Oracle 서버의 분석과 AI 서버의 RAG 파이프라인에 필요한 원시 데이터를 공급
    - 필요성
        - Oracle 서버의 LSTM Autoencoder는 동일 장비의 과거 Lot 시계열 데이터가 축적되어야 트렌드 이상 감지가 가능하므로 장기 이력 저장소가 필요
        - RDBMS 대신 TSDB를 사용하여 대용량 시계열 데이터의 쓰기 성능과 시간 범위 쿼리 성능을 확보
    - 데이터 흐름: 가상 EAP 서버(비전 머신) ==> Broker 서버 ==> Historian 서버   
    - 플렛폼: Node.js(TypeScript)
    - DB: Postgres TimescaleDB(TBMS), InfluxDB
- 가상 EAP 서버
    - 설명: 실제 비전 검사 장비(EAP)를 대체하는 Mock Publisher 서버로, 검사 결과 데이터를 JSON 포맷으로 직렬화하여 MQTT Broker에 발행
    - 목적: 실제 장비 없이 전체 데이터 파이프라인을 검증할 수 있는 테스트 환경을 제공합니다. N대의 장비 시나리오(PASS / FAIL / MARGINAL)를 재현하여 모바일 앱, Oracle 서버, Historian 서버를 병렬로 테스트함
    - 언어: C#
- MES 서버
    - 설명: N대의 가상 비전 머신을 MQTT를 통해 중앙에서 제어(Lot 시작, 상태 변경, 레시피 변경)하고, Lot End 정보를 저장하는 생산 제어 서버
    - 목적 
        - 테스트 환경에서 각 장비를 개별 조작하는 대신 하나의 MES에서 모든 가상 비전 머신을 일괄 제어하여 QA 사이클을 단축
        - 레시피별 성공/실패 이력과 Oracle 서버의 기기 이상 징후를 연계하여 오퍼레이터의 휴먼 에러를 사전에 방지
    - 필요성
        - N대 장비를 개별 수동 조작하는 방식은 테스트 시나리오 재현이 어렵고 비효율적이므로 중앙 일괄 제어가 필요
        - Oracle 서버에서 이상 징후가 감지된 기기에 작업 요청이 들어올 경우 사전 경고 없이는 오퍼레이터가 인지할 방법이 없음
        - 실패 이력이 있는 레시피를 오퍼레이터가 선택할 경우 경고를 제공하여 동일한 실수 반복 방지
    - 데이터 흐름: MES <==> Broker(MES 내부) <==> 가상 EAP 서버
    - 언어: C#
- Dispatcher 서버
    - 설명: Local Area의 TSDB에서 read-only로 데이터를 조회한 후, 데이터를 비식별화 처리하여 Online Area의 AI 서버로 단방향 Push하는 데이터 게이트웨이 서버
    - 목적: Local Area와 Online Area 사이의 보안 경계를 유지하면서 시계열 데이터를 AI 서버로 안전하게 전달합니다. 비식별화 처리를 통해 민감한 공정 데이터의 외부 유출을 방지
    - 필요성
        - AI 서버(Online Area)가 Broker나 Historian에 직접 접근하면 Local Area 핵심 자산이 외부에 노출되어 보안 위협이 발생하므로 중간 게이트웨이가 필요
        - Dispatcher는 TSDB에 read-only 권한만 가져 최소 권한 원칙을 준수하고, 단방향 데이터 흐름으로 침투 경로를 차단 (DMZ 구성)
        - AI 서버가 TSDB를 직접 Polling하면 불필요한 읽기 부하가 발생하므로 Dispatcher가 배치 단위로 부하를 격리
    - 데이터 흐름: TSMS ⇒ Dispatcher 서버 ⇒ AI 서버
    - 플렛폼: Node.js(TypeScript)
- Oracle 서버(후순위 개발 예정)
    - 설명: 현재 Lot의 즉각적인 MARGINAL 판정(1차)과 장비의 점진적 열화 패턴 감지(2차)를 수행하는 Local Area 내 Rule-Based & AI 판정 엔진
    - 목적
        - 성공적으로 완료된 Lot 중에서 성공과 실패 사이의 회색지대(MARGINAL, Fail기준 80~95% 사이)에 있는 데이터를 파악하여 오퍼레이터에게 경고 메시지를 전달
        - 단순 수치 증가 추세는 정상 마모로 간주하되, 증가 패턴 자체가 갑자기 달라지는 순간을 이상으로 감지하여 해당 기기의 신규 작업을 사전에 차단
    - 필요성
        - EAP가 FAIL을 직접 판정하지만, FAIL 직전 경계 구간(MARGINAL)은 별도 검증 레이어가 필요
        - 경계 구간의 정의가 변경되어도, 모바일 기기에 별도의 업데이트 없이 모든 모바일 기기에 동일한 기준으로 메시지 전달 (Loose Coupling)
        - 단순 수치 임계값으로 잡히지 않는 점진적 장비 열화 중, 급격한 절삭 품질 악화와 같은 이상 패턴을 사전에 포착하여 예지 보전 실현
    - 언어: Python
    - 1차 검증
        - 방식: rule-based
        - 목적
            - 가상 EAP 서버에서 발행한 lot 데이터를 검증하여 PASS 중에서 MARGINAL 구간 여부 파악
            - 임계점에서 벗어난 절삭 품질과 열화된 조명 상태를 가진 장비를 파악
        - 데이터 흐름: 가상 EAP 서버 => Broker 서버 => Oracle 서버 => Broker 서버 => mes 서버, 모바일 기기 (pass시)
    - 2차 검증
        - 방식: local AI, EWMA + MAD / Isolation Forest 모델
        - 목적
            - LOT_END 이벤트 수신 시 레시피별로 독립 관리되는 과거 LOT 시계열 데이터를 분석하여 동적 주의 경계선 산출
            - 단순 Pass/Fail 이분법에서 벗어나 정상과 Fail 사이의 주의 단계 유닛을 검출하여 오퍼레이터에게 사전 경보 제공
            - 레시피 전환 시 해당 레시피의 LOT 이력만 참조하여 기준을 독립적으로 학습, 오경보 없는 신규 레시피 대응
            - 산출된 동적 임계값을 오퍼레이터 승인 후 1차 Rule DB에 자동 반영하여 코드 수정 없이 검증 기준 갱신
            - LOT 완료 시 수율·주된 불량 원인·주의 유닛 요약이 담긴 LOT 보고서를 자동 생성하여 모바일로 전송
        - 데이터 흐름: Broker 서버(LOT_END) => Oracle 서버 
        => Rule DB 업데이트 => Broker 서버 => MES 서버, 모바일 기기
        - DB: Postgres TimescaleDB (LOT 시계열 · 판정 이력 · Rule 이력)

## LLM을 이용한 자동 리서치 보고서 작성
### 문제 인식
- 통합 관제 가시성 결여: 특정 구역 내 다수 설비에서 발생하는 상호 연관된 퍼포먼스 지표와 이벤트를 단일 관점에서 통합 모니터링하고 분석할 수 있는 고도화된 솔루션이 부재합니다.
- 분석 생산성의 한계: EDA 등을 통해 수집되는 방대한 시계열 및 로그 데이터를 소수의 분석가가 수동으로 분석하는 방식은 막대한 시간과 리소스가 소요되며, 실시간 이상 징후 포착 및 예지 보전에 한계가 있습니다.
- 데이터 분석의 병목 현상: 수만 건의 비정형 로그 속에서 유의미한 상관관계를 찾아내는 과정에서 분석가의 숙련도에 따라 통찰의 질이 결정되는 인적 의존성 문제가 발생합니다.

### 해결 방안
- RAG(검색 증강 생성) 기반 지능형 분석: Historian 서버(TSDB)에 적재된 시계열 로그와 설비 매뉴얼을 벡터 데이터베이스에 임베딩하여, 현재 구역의 상태를 다각도로 분석하는 RAG 파이프라인을 구축합니다.
- LLM 기반 자동 리서치 리포팅: 복잡한 수치와 텍스트 로그를 LLM이 실시간으로 요약 및 분석하여, 구역의 효율성 리포트와 권고안을 인간이 이해하기 쉬운 자연어 형태로 자동 생성합니다.
- 의사결정 최적화: 분석가는 원시 데이터를 뒤지는 대신 AI가 생성한 인사이트를 검토하고 최종 의사결정만 내림으로써, 공정 관리의 효율성을 극대화합니다.
### 필요한 서버 종류
- AI 서버
    - 설명: Dispatcher 서버로부터 수신한 시계열 데이터를 전처리 후 벡터 임베딩하여 Vector DB에 적재하고, 웹 백엔드 서버의 요청 시 RAG + CAG 방식으로 분석 데이터를 제공하는 지능형 분석 서버
    - 목적 
        - Historian TSDB에 적재된 시계열 로그와 설비 매뉴얼을 벡터 데이터베이스에 임베딩하여 RAG 파이프라인을 구축
        - LLM이 복잡한 수치와 텍스트 로그를 자연어 인사이트로 변환하여 구역별 효율성 리포트와 권고안을 자동 생성
    - 필요성
        - 수만 건의 비정형 로그를 소수 분석가가 수동 분석하는 병목 현상을 해소하기 위해 필요
        - 분석가의 숙련도에 따라 통찰의 질이 달라지는 인적 의존성 문제를 LLM 기반 자동화로 해소
        - 분석가는 원시 데이터를 뒤지는 대신 AI가 생성한 인사이트를 검토하고 최종 의사결정만 수행하여 공정 관리 효율성 극대화
    - 데이터 흐름: Historian(TSDB) ⇒ Dispatcher 서버 ⇒ AI 서버
    - 플렛폼: Python
    - 방식: RAG, CAG
    - 모델: Claude 4.6 Sonnet
    - DB: Postgres pgvector(Vector DB), Chroma DB
