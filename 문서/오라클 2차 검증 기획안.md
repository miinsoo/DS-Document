## 1. 프로젝트 주요 목표

반도체 후공정 비전검사 현장에서 오퍼레이터는 매 LOT마다 수백~수천 개의 유닛 검사 결과를 수동으로 확인해야 한다. 이 과정에서 두 가지 핵심 문제가 발생한다.

1. 방대한 데이터를 사람이 일일이 검토하는 것은 현실적으로 불가능하다. 

2. 단순 Pass/Fail 이분법으로는 회색지대에 있는 유닛을 사전에 포착하기 어렵다.  

이 두 문제를 해결하기 위해 Oracle 서버의 2차 검증 시스템을 기획한다.

### 1-1. AI 기반 자동 보고서 생성

LOT가 완료될 때마다, 그리고 일별·주별 주기로 AI가 데이터를 분석해 자연어 요약 보고서를 자동 생성한다. 오퍼레이터는 수백 개의 유닛을 직접 확인하는 대신 보고서 한 장으로 LOT의 품질 상태를 파악할 수 있다.

| 보고서 종류 | 주기 | 채널 | 주요 내용 |
|------------|------|------|----------|
| LOT 보고서 | LOT 완료 시 즉시 | 모바일 | 수율, 주된 불량 원인, 주의 유닛 요약, 권고사항 |
| 일일 보고서 | 매일 종료 시 | 웹 | 당일 전체 LOT 통계, 이상 발생 현황, 트렌드 |
| 주간 보고서 | 매주 월요일 | 웹 | 주간 수율 추이, 레시피별 비교, 개선 권고 |

### 1-2. Pass/Fail 중간 구간(주의 단계) 유닛 검출

기존 시스템은 Pass와 Fail만 구분한다. 그러나 실제 현장에서는 기준을 통과했지만 경계에 가까운 유닛, 즉 "지금은 Pass지만 다음 LOT에서 Fail로 전환될 가능성이 있는 유닛"이 존재한다. 이 구간을 **주의 단계**로 정의하고 별도로 검출한다.

```
기존:  [ FAIL ]──────────────────[ PASS ]

개선:  [ FAIL ]──[ 위험 ]──[ 주의 ]──[ 정상 ]
                    ↑
            이 구간을 AI가 감지(주의 단계 유닛 과다)
```

주의 단계 유닛이 많아지는 추세가 감지되면 실제 Fail이 급증하기 전에 오퍼레이터에게 경보를 보낸다.

## 2. 문제 인식 — 1차 Rule-based 검증의 한계

현재 1차 검증은 고정된 임계값으로 Pass/Fail을 판정한다. 이 구조는 세 가지 근본적인 문제를 가진다.

### 2-1. 임계값 업데이트 문제

고정 임계값은 장비 상태나 현장 환경이 변해도 기준이 바뀌지 않는다. 장비가 정상적으로 노화되거나 소모품이 교체된 후에도 예전 기준이 그대로 적용되어 불필요한 경보가 반복되거나, 반대로 실제 이상을 놓치게 된다.

### 2-2. 레시피별 관리 문제

레시피마다 검사 대상 패키지의 크기, 형상, 공차 기준이 다르다. 그러나 1차 검증은 모든 레시피에 동일한 임계값을 적용한다.

### 2-3. 셋업 시 과도한 이상치 문제

장비 최초 도입 시 Teaching 구간에서는 정상 양산보다 Fail과 Exception이 훨씬 많이 발생한다. 이 기간의 데이터를 기준 학습에 그대로 사용하면 오염된 임계값이 생성된다.

## 3. 해결 방안

### 3-1. 레시피별 독립적인 동적 경계 학습

임계값을 고정하는 대신 **레시피별로 독립적으로 데이터를 축적하고 경계를 자동 학습**한다.

레시피가 전환될 때마다 해당 레시피의 LOT 이력만 참조해 정상/주의/위험 경계를 계산한다. 다른 레시피의 데이터는 일절 참조하지 않는다. LOT가 쌓일수록 경계가 점점 정교해지는 구조다.

```
레시피 전환 감지
        ↓
해당 레시피의 LOT 이력 조회
        ↓
LOT 5개 미만:    초기 시딩 기준 사용
LOT 5개 이상:    실측 데이터로 경계 자동 계산
LOT 10개 이상:   복합 이상 감지 모델 추가 활성화
```

**오퍼레이터 초기값 시딩**

셋업 구간 오염 문제를 해결하기 위해 레시피 최초 투입 시 오퍼레이터가 "이 레시피의 예상 정상 수율 범위"를 직접 입력한다. 시스템은 이 값을 시작점으로 삼아 실제 LOT 데이터로 점진적으로 보정한다. 초기 셋업 구간의 이상치 영향이 데이터가 쌓일수록 자동으로 희석된다.

### 3-2. LOT별 분석 보고서와 오퍼레이터 승인 구조

**LOT 보고서**

LOT가 완료될 때마다 다음 내용을 자동 생성해 모바일로 전송한다.

- 수율 및 정상 범위 대비 위치
- 주된 불량 원인 (ErrorType 분포 기반)
- 주의 단계 유닛 수 및 비율
- AI 분석 코멘트 및 권고사항
- 제안된 근거 설명(블랙박스 제공)
- 현재 기준이 몇 개 LOT 학습값인지 투명하게 표시

**오퍼레이터 승인 구조**

AI가 기준 조정을 제안할 수 있지만 **자동 적용하지 않는다.** 모든 임계값 변경은 오퍼레이터의 수락/거부를 거친다. 이는 두 가지 문제를 방지한다.

1. 불량 LOT가 연속으로 들어올 때 불량 상태를 "정상"으로 학습하는 것을 막는다.  
2. 현장 엔지니어의 도메인 지식이 AI 판단을 보정할 수 있는 구조를 유지한다.

```
AI 기준 조정 제안
        ↓
모바일 알림 (제안 근거 포함)
        ↓
오퍼레이터 수락 → 즉시 적용
오퍼레이터 거부 → 기존 기준 유지  
```

## 4. 설계

### 4-1. 1차 검증과 2차 검증의 관계

실시간성을 유지하면서 임계값도 개선하기 위해 두 검증을 **비동기로 분리**한다.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[실시간 경로]          오퍼레이터 즉시 확인용(1차 검증)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LOT_END
        ↓
  1차 검증 (Rule DB에서 현재 임계값 조회)
        ↓
  즉시 오퍼레이터에게 전달

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[비동기 경로]          LOT 완료 후 임계값 개선용(2차 검증)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LOT_END 
        ↓
  2차 검증 (EWMA+MAD로 새 임계값 계산)
        ↓
  LOT 보고서 + 임계값 변경 제안 전송
        ↓
  오퍼레이터 승인 → Rule DB 자동 업데이트
        ↓
  다음 INSPECTION_RESULT부터 새 임계값 적용
```

두 경로는 독립적으로 동작한다. 2차 검증이 진행 중이어도 1차 검증의 실시간 판정에는 영향을 주지 않는다.

### 4-2. Rule DB

1차 검증의 임계값을 코드 안에 하드코딩하지 않고 **DB 테이블에서 읽어오는 구조**로 설계한다. 오퍼레이터가 승인하면 DB 값만 바뀌고, 1차 검증은 다음 유닛부터 자동으로 새 기준을 적용한다.

**Rule 테이블 구조**

| recipe_id | metric | warning_threshold | danger_threshold | updated_at | approved_by | lot_basis |
|-----------|--------|-------------------|-----------------|------------|-------------|-----------|
| Carsem_3X3 | yield_pct | 94.5 | 92.0 | 2026-01-22 | ENG-KIM | 28 LOT 기반 |
| Carsem_4X6 | yield_pct | 65.0 | 60.0 | 2026-01-27 | ENG-KIM | 8 LOT 기반 |
| Carsem_3X3 | et52_rate | 3.0 | 8.0 | 2026-01-22 | ENG-KIM | 28 LOT 기반 |
| Carsem_3X3 | et30_rate | 1.5 | 4.0 | 2026-01-22 | ENG-KIM | 28 LOT 기반 |

**Rule 이력 테이블 구조**

임계값이 변경될 때마다 이전 값과 변경 사유를 자동으로 기록한다. 언제, 누가, 어떤 근거로 기준을 바꿨는지 추적이 가능하다.

| changed_at | recipe_id | metric | old_value | new_value | approved_by | ai_basis |
|------------|-----------|--------|-----------|-----------|-------------|----------|
| 2026-01-22 | Carsem_3X3 | yield_pct | 90.0 | 94.5 | ENG-KIM | 28 LOT EWMA μ=96.8%, σ=0.9% |
| 2026-01-27 | Carsem_4X6 | yield_pct | 90.0 | 65.0 | ENG-KIM | 8 LOT EWMA μ=68.2%, σ=4.1% |

### 4-4. 모델

#### 모델 종류

**EWMA + MAD (주 모델)**

EWMA로 최근 LOT일수록 더 높은 가중치를 부여한다. 오래된 셋업 데이터의 영향이 시간이 지날수록 자동으로 줄어든다. MAD로 단순 표준편차보다 이상치에 강건하게 반응하게 한다. 셋업 구간의 극단적인 Fail LOT가 포함되어도 기준선이 크게 흔들리지 않는다.

```
활성 조건: 레시피별 LOT 5개 이상
역할:      레시피별 정상/주의/위험 경계선 동적 계산
산출물:    3구간 경계값 (정상·주의·위험)
```

**Isolation Forest (보조 모델)**

단일 지표로는 정상이지만 여러 지표를 동시에 봤을 때 조합이 비정상인 LOT를 감지한다. 각각의 지표는 Rule-based를 통과해도 수율, 불량률, 소요시간, ET 분포가 동시에 나쁜 방향으로 이동하면 이상으로 판정한다.

```
활성 조건: 레시피별 LOT 10개 이상
역할:      복합 지표 이상 감지 (단일 지표 감지 보완)
산출물:    이상도 점수 0~1 (0.5 이상부터 주의)
```

### 4-3. 판정 3단계 구조

```
정상  │ EWMA 정상 범위 내 + Isolation Forest 점수 < 0.5
      │ → 보고서 생성, 경보 없음
──────┤
주의  │ EWMA 2σ 초과 또는 Isolation Forest 점수 0.5~0.85
      │ → 보고서 + 모바일 주의 알림 + 오퍼레이터 확인 요청
──────┤
위험  │ EWMA 3σ 초과 또는 Isolation Forest 점수 0.85 초과
      │ → 보고서 + 즉시 경보 + 작업 중단 권고
```


## 5. 시스템 제약 및 한계 인정

모든 시스템 설계에서 한계를 명확히 인식하는 것이 중요하다.

| 한계 | 내용 | 대응 방안 |
|------|------|----------|
| 초기 데이터 오염 | 셋업 구간 데이터가 양산 기준과 달라 초기 임계값 부정확 | 오퍼레이터 시딩으로 초기값 보정 |
| 소량 데이터 | LOT 10개 미만 시 복합 감지 모델 비활성 | 보고서에 "학습 중" 상태 명시 |
| 설비 직접 감지 불가 | 블레이드 마모 등 직접 지표 없음 | 간접 지표(ET 분포, 소요시간)로 추론 |
| 오퍼레이터 의존성 | 초기 시딩값 품질에 따라 초기 정확도 차이 | 시딩 입력 가이드라인 및 디폴트값 제공 |

## 6. 기대 효과

**오퍼레이터 업무 효율화**  
LOT당 수백 개 유닛을 수동 확인하는 작업이 자동 요약 보고서 한 장으로 대체된다.

**사전 예방적 품질 관리**  
Pass/Fail 이분법에서 벗어나 주의 단계를 조기에 감지함으로써 대규모 Fail 발생 전에 조치가 가능해진다.

**레시피 전환 리스크 감소**  
신규 레시피 투입 시 독립적인 기준 학습이 시작되어 오경보 없이 안정적인 초기 양산이 가능해진다.

**데이터 기반 의사결정**  
엔지니어의 경험과 직관에 의존하던 임계값 설정이 실측 데이터 기반으로 전환된다.
